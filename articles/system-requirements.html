Sum of GCD of Formed Pairs

Introduction

Many array-based algorithmic problems become much easier once they are broken down into smaller, independent steps. The Sum of GCD of Formed Pairs problem is an excellent example of this approach. Although the problem statement initially appears complex because it involves constructing a new array, sorting it, and forming pairs in a specific order, the solution itself is straightforward once each stage is understood.

The problem combines several important concepts frequently encountered in competitive programming and technical interviews:

  • Running Maximum
  • Greatest Common Divisor (GCD)
  • Prefix Processing
  • Sorting
  • Pair Formation
  • Complexity Analysis

In this article, we will understand the problem step by step, develop an efficient algorithm, prove its correctness, analyze its complexity, and finally examine the code implementations.

Problem Statement

Given an integer array nums of length n, construct another array called prefixGcd.

For every index i:

mx_i = max(nums[0], nums[1], ..., nums[i])

Then compute:

prefixGcd[i] = gcd(nums[i], mx_i)

After constructing the prefixGcd array:

  1. Sort it in non-decreasing order.
  2. Pair the smallest element with the largest element.
  3. Pair the second smallest with the second largest.
  4. Continue until no more pairs can be formed.
  5. Compute the GCD of every pair.
  6. Return the sum of these GCD values.

If the array contains an odd number of elements, the middle element remains unpaired and is ignored.

Example

Input: nums = [3, 6, 2, 8]

Step 1: Construct the Prefix GCD Array

First, compute the running maximum.

Index nums[i] Running Maximum
033
166
226
388

Now compute the GCD for every position.

Index Running Maximum prefixGcd
033
166
262
388

Therefore, prefixGcd = [3, 6, 2, 8].

Step 2: Sort

After sorting: [2, 3, 6, 8]

Step 3: Form Pairs

Pair the smallest with the largest:

  • 2  ←→  8
  • 3  ←→  6

Step 4: Compute Pair GCD

gcd(2, 8) = 2
gcd(3, 6) = 3

Total: 2 + 3 = 5

Hence, the output is 5.

Understanding the Concepts

Running Maximum

One important observation is that the maximum element up to the current index can be maintained efficiently. Instead of searching from the beginning every time, we store a variable called mx.

Initially: mx = 0
For every element: mx = max(mx, nums[i])

Notice that the running maximum never decreases.

Why Do We Sort?

The problem requires pairing the smallest elements with the largest elements. Sorting places the smallest values at the beginning and the largest values at the end, making the required two-pointer pairing technique extremely clean. We simply move inward from both ends.

Algorithm Flow

1. Raw Input
Accept the base numerical unsorted input array (nums).
2. Running Maximum Extraction
Track and map incremental peak boundaries dynamically.
3. Prefix GCD Compilation
Compute element values using their index paired against local maximum elements.
4. Numerical Sorting
Organize values linearly to isolate relative scale boundaries.
5. Extremes Pairing
Implement a dual inward pointer movement pattern (Smallest ↔ Largest).
6. Component GCD Resolution
Calculate sub-divisors for every isolated coordinate pair.
7. Sum Aggregation
Consolidate all values into a singular unified solution metric.

Pseudocode

// Phase 1: Build prefixGcd
mx = 0
for i = 0 to n - 1:
    mx = max(mx, nums[i])
    prefixGcd[i] = gcd(nums[i], mx)

// Phase 2: Sort
sort(prefixGcd)

// Phase 3: Pair and Calculate
answer = 0
for i = 0 to (n / 2) - 1:
    answer += gcd(prefixGcd[i], prefixGcd[n - 1 - i])

return answer

Correctness Proof

Lemma 1

The running maximum maintained by the algorithm always equals max(nums[0], ..., nums[i]).

Proof: At every iteration, we update mx = max(mx, nums[i]). Therefore, after processing index i, mx stores the maximum value seen from index 0 through i. Hence, prefixGcd[i] = gcd(nums[i], mx) is computed exactly as defined.

Lemma 2

After sorting, the algorithm forms the required pairs.

Proof: Sorting produces an ordered sequence. The algorithm pairs prefixGcd[i] with prefixGcd[n - 1 - i], which naturally moves inward from both boundaries, matching the requested pairing logic perfectly.

Complexity Analysis

  • Constructing prefixGcd: Requires a single traversal taking O(n) time.
  • Sorting: Dominates the runtime, taking O(n \log n) time.
  • Pair Computation: Iterates through half the array, taking O(n) time.

Overall Time Complexity: O(n \log n)

Space Complexity: O(n) due to the allocation of the prefixGcd array.

Code Implementations

class Solution {
public:
    long long gcdSum(std::vector<int>& nums) {
        int n = nums.size();

        std::vector<int> prefixGcd(n);
        int mx = 0;

        // Phase 1: Compute Running Max and GCD
        for (int i = 0; i < n; i++) {
            mx = std::max(mx, nums[i]);
            prefixGcd[i] = std::gcd(nums[i], mx);
        }

        // Phase 2: Sort elements
        std::sort(prefixGcd.begin(), prefixGcd.end());

        long long ans = 0;

        // Phase 3: Move inward from both ends
        for (int i = 0; i < n / 2; i++) {
            ans += std::gcd(prefixGcd[i], prefixGcd[n - i - 1]);
        }

        return ans;
    }
};
import math

class Solution:
    def gcdSum(self, nums: list[int]) -> int:
        n = len(nums)
        
        prefix_gcd = []
        mx = 0
        
        # Phase 1: Compute Running Max and GCD
        for x in nums:
            mx = max(mx, x)
            prefix_gcd.append(math.gcd(x, mx))
            
        # Phase 2: Sort elements
        prefix_gcd.sort()
        
        ans = 0
        
        # Phase 3: Move inward from both ends
        for i in range(n // 2):
            ans += math.gcd(prefix_gcd[i], prefix_gcd[n - i - 1])
            
        return ans

Key Takeaways

  • Maintain a running maximum inline rather than re-scanning to achieve linear performance during data generation.
  • Sorting completely simplifies symmetric processing problems (like pairing extremes).
  • The algorithm naturally drops the odd element out because n / 2 dynamically drops fractions during integer division.
